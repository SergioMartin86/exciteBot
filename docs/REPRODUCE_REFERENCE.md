# Reproducing the reference TAS with `jaffar-player` (instructions for the new-session Claude)

This explains how to replay the lap-1 reference TAS on the **real NES emulator** and how to (re)generate the
`tas.ram` validation harness. **All commands below are verified working** (run on this machine, 2026-06-26).

The kit in `reference/` is **self-contained** — it bundles the correct prebuilt player binary, the ROM, the
config, the raw input movie, and the boot sequence. You do **not** need the old jaffarPlus repo to run it.

```
reference/
  jaffar-player              # CORRECT prebuilt binary (raw-input replay; do not substitute — see GOTCHAS)
  race01.jaffar              # JaffarPlus config (QuickerNES; win = Race Over Flag (0x52) > 0)
  race01.sol                 # the reference movie: 2262 lines of RAW per-frame NES inputs (full 4-loop race)
  race01.initial.sol         # boot/countdown sequence (REQUIRED by the config; runs before the movie)
  Excitebike (JU) [!].nes    # ROM (SHA1 2E9897846E54A4A9865E87DE7517C6710BDEC255), gitignored
```

> **Filenames:** this kit uses `race01.*` (the old project used `race01a.*`). The `jaffar-player` binary and
> `tas.ram` are **gitignored** (binary / large RAM dump); the binary is recoverable from
> `/home/jaffar/jaffarPlus/build/jaffar-player` and `tas.ram` is regenerated by §2 below.

## 1. Reproduce the reference (sanity check: should WIN at frame 2261)

Run **from inside `reference/`** (the config uses relative paths for the ROM and boot sequence):

```bash
cd reference
./jaffar-player race01.jaffar race01.sol \
    --printFinalState --disableRender --unattended --exitOnEnd 2>&1 \
  | sed 's/\x1b\[[0-9;]*[A-Za-z]//g' \
  | grep -aoE '(Current Step:[ 0-9]*[0-9]+|Race Over Flag:[ 0-9]*[0-9]+|Bike Pos X:[ 0-9]*[0-9]+\.[0-9]+ \([0-9 ,]+\))' | tail -3
```

**Expected output (the pass condition):**
```
Race Over Flag:01
Current Step:2262
Bike Pos X:12157.328 (125, 84)
```
The `Race Over Flag (RAM[0x52])` flips to 1 at frame **2261** = the race completes; the movie plays out to its
last input at step 2262 with `Bike Pos X = 12157.328`. `(125, 84)` are `RAM[0x50]=125`, `RAM[0x394]=84`.

> The `sed` strips the player's ncurses escape codes (this binary renders an interactive display even with
> `--unattended`; `--exitOnEnd` makes it run to the end and quit). The replay itself is correct; the `sed`
> just cleans the captured text.

## 2. (Re)generate the `tas.ram` validation harness

`tas.ram` is gitignored (large binary); regenerate it into the repo root with:

```bash
cd reference
./jaffar-player race01.jaffar race01.sol \
    --dumpRam ../tas.ram --disableRender --unattended --exitOnEnd
```

`--dumpRam` writes the NES low RAM (`$0000–$07FF`, 2048 bytes) **once per frame**, concatenated:
**2263 frames × 2048 bytes = 4,634,624 bytes**. Frame `f`, byte `addr` → offset `f*2048 + addr`.
This is the file your native model must reproduce byte-for-byte.

## 3. How to validate your native model against the real game

1. Read the raw inputs from `reference/race01.sol` (one NES input per line, frames 0..2261; the bike is
   already past the boot countdown — `race01.initial.sol` handles that, and `tas.ram` frame 0 is the first
   post-boot frame).
2. Feed those inputs into your model and dump your model's RAM (or just the modeled variables) per frame.
3. Assert your per-frame values equal `tas.ram` at the addresses in the handoff doc's variable map
   (velX16 `0x90/0x94`, posX `0x50`+`0x394`, airMode `0xB0`, angle `0xAC`, slope `0xD4`, posY `0x8C`,
   temp `0x3B6`, Race Over `0x52`, Current Loop `0x3A4`, …). Iterate until **all 2263 frames match exactly**.
   Only then start searching.

## 4. GOTCHAS (these cost time in the old project — read them)

- **Use the `reference/jaffar-player` binary as-is** (gitignored; recover from
  `/home/jaffar/jaffarPlus/build/jaffar-player`). It was built from the *raw-input-mode* `exciteBike.hpp`.
  Other old build dirs (e.g. `build-eb-baseline`) embed a *commit-token-decoding* version of the game logic
  that **silently ignores the raw `.sol` inputs and diverges**. If you ever rebuild the player from the old
  jaffarPlus repo, make sure `race01.jaffar` carries **no** `"Commit And Hold"` / `"Airborne Per Frame Angle"`
  flags (absent → the game applies inputs raw → faithful replay).
- **No trailing blank line in the `.sol` files.** A trailing empty line makes the input parser throw
  `Could not decode input string: ''` (`inputParser.hpp:149`) and the player aborts. Strip it if you re-export.
- **`race01.sol` is RAW inputs**, not commit tokens (no `S`/Start markers). It only replays faithfully
  through the raw-mode game logic.
- **`race01.initial.sol` is required** — it's the boot/countdown the config plays before the movie. Keep it
  next to the config.
- The player needs standard system libs (`libSDL2`, `libncurses`, `libstdc++`, …); all resolve on this
  machine. `--disableRender` avoids needing a display.
- The ROM path in `race01.jaffar` is **relative** (`Excitebike (JU) [!].nes`), so always run from
  `reference/`.

## 5. Useful `jaffar-player` flags (reference)

`./jaffar-player <config.jaffar> <solution.sol> [flags]`
- `--printFinalState` — print the full game state each step (incl. `Bike Pos X`).
- `--dumpRam <path>` — write per-frame NES RAM (the harness generator).
- `--dumpHashes <path>` — write per-frame state hashes.
- `--disableRender` — no SDL window.
- `--unattended` — don't wait for interactive keypresses.
- `--exitOnEnd` — quit at the last step (otherwise it idles at the end).
- `--saveStateStep N --saveStateFile <path>` — dump a savestate at step N (useful for seeding).
