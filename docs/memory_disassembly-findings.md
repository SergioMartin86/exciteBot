Reverse-engineering of Excitebike (NES) to beat the lap-1 reference TAS. **Annotated disassembly (with all findings inline + header summary): `examples/nes/exciteBike/excitebike_FF.commented.asm`** (copy of the cyneprepou4uk bank_FF.asm). Raw source also at scratchpad/excitebike_FF.asm. Findings VALIDATED vs the TAS RAM dump (scratchpad/tas.ram, 1469 frames, 2048 B/frame).

**Variable map** (player=obj index 0; "obj" arrays 4-wide): 0x4C player index (CPX ram_004C gates player physics); 0x52 track_finished_flag; 0x50 scroll_X=posX low px; 0x5C THROTTLE (bit7 0x80=A slow, bit6 0x40=B turbo/fast); 0x90 velX low, 0x94 velX high => **velX16 = 0x94*256 + 0x90**; 0x98 crash; 0xAC angle; 0xB0 airMode(0 ground/2 air); 0xB8 posZ2; 0x394 posX subpixel; 0x3B6 ENGINE TEMP.

**Speed model**: sub_CE29 (player accel) velX += tbl_C0BC[Y], clamp to (tbl_C0D1[Y]hi:tbl_C0CE[Y]lo); sub_CE58 friction (-tbl_C0C1[Y]). Y by throttle: A(bit7)->Y0 accel24 cap **0x0320=800**; B(bit6)->Y1 accel63 cap **0x0340=832** (FAST); track_finished->Y2 cap 0x017F=383; airborne->friction only (NO accel in air). Tables@C0BC: accel[24,63,40,32,40], friction[56,12,0,60,28,192,127], maxlo[0x20,0x40,0x7F], maxhi[0x03,0x03,0x01].

**What the TAS does** (tas.ram): flat cruise velX=832 = EXACTLY the B(Y1) cap -> rides the cap. Throttle over 1469 frames: B(0x40)=688, coast(0x00)=780, A(0x80)=**0** (never), 0xC0=1. So TAS uses ONLY B+coast; coasts to manage heat. velX rises to 1088@~1080 then **1544**@1200-1468 (finish) -> EXCEEDS flat cap via DOWNHILL/gravity. Engine temp 0x3B6 stays 8..29 (never overheats in the dump).

**TEMPERATURE model COMPLETE** (E36B-E3C0): per-throttle equilibrium temp tbl_D8FB[Y]=[coast 0x08=8, B 0x20=32, A 0x11=17]; heat/approach rate tbl_D8F7[Y]=[coast 63, B 15, A 7] (added to subcounter 0x3B5; carry steps temp 0x3B6). **OVERHEAT STALL at temp>=0x20=32** (sets 0x3E0 stall obj + timer 0x3C); danger warn at >=0x1A=26 while holding B. KEY: **B equilibrium = 32 = the stall threshold**, so holding B drives temp TO stall -> B is self-limiting, can't be held forever; coast equilib=8 cools fast (rate 63).

**TAS temperature behavior (tas.ram)**: NEVER ground-coasts >=4 frames (0 such runs) => on the GROUND it holds B essentially ALWAYS (pinned at the 832 cap); the 780 'coast' frames are the AIRBORNE/jump frames (throttle does nothing airborne AND coasting there cools toward 8). So the jumps themselves provide the cooling that keeps B held on the ground. Temp self-manages via jump cadence (peaks 29, never stalls).

**COLLISION/TERRAIN study (no exploitable oversight in race01)**: GROUND terrain-following (E7A9-E7C8): each ground frame reads track-data nibble (AND #$0F), FORCES bike angle 0xAC = nibble, slope 0xD4 = tbl_E6AD[angle-2]. => player controls angle ONLY in the air (ground angle = terrain). LANDING crash (DC51-DC6D): safe iff tbl_D86C[Y] <= angle(0xAC) < tbl_D87C[Y], Y = slope(0xD4) + (8 if velX_high<2 else 0). Windows (lo/hi per slope, fast band): slope0=[3,12) slope1=[2,9) ... slope7=[0x0F,0x00)=EMPTY=forced-crash; slow band (slope+8) slightly WIDER (more forgiving). Perfect landing (angle==tbl_D88B[slope] target -> 0x368) skips the wobble/recovery (airMode->1 at DC7B) => keeps speed; the posX search finds this. LATENT OOB (not reachable in race01): terrain nibble 0 or 1 -> angle-2 underflows Y to 0xFE/0xFF -> tbl_E6AD OOB read for the slope. But TAS ground angles are always 2..11 (flat=6, 607 frames), NEVER 0/1, so race01 never triggers it. No input-exploitable terrain/collision glitch found; physics are clean.

**LAUNCH/AIR/DOWNHILL MODEL (corrected 2026-06-26 from tas.ram dynamics)**: TAKEOFF sub_DD38 (terrain type-06 handler ofs_002_E934 -> sub_DD06 -> DD38): sets airMode=2, 0x380=0x0F, and **launch velZ (0x378/0x37C) = velX16 + 0x00AF(175)** (faster entry => higher/longer arc); HALVED if 0x388==2 (only after a wobble landing, DC81). Min-speed gate: velX_hi==0 => no launch. 0x388(launch type) set at E94A = 1 iff SLOW else 0. DESCENT integrator sub_DD6F: 0x380/0x384 += 0x38C where **0x38C = tbl_D868[0x5C&3]** = {idx0:52, idx1:52, idx2:24, idx3:52} -> the in-air LEAN (low 2 bits of 0x5C) shapes the arc/landing-angle (search varies this per frame). **BIG CORRECTION to the old "land sooner = more posX" premise (it was BACKWARDS)**: airborne velX is **CONSTANT** (player air-friction ~0): every jump holds its takeoff speed exactly (832->832 ... 1544->1544). Air velX mean 1152 (min 832 = the ground cap, max 1544) vs GROUND mean 810 -> **flying is >= ground, usually FASTER**, never slower. TAS is **airborne 50.2%** (737/1469, 9 jumps, runs up to 143f) precisely because ramps are unavoidable AND air maintains speed at no cost. velX timeline: pinned 832 on the FLAT (f0-1063, hard cap, 1006 frames ==832), then a DOWNHILL accelerates 832->1088->1344->**1544** (f1063-1196) held 273f to the finish. So 1544 is the downhill terminal velocity for this track.

**CONCLUSION / where a beat can come from (revised)**: NO slack on the flat (832 = hard B cap, held) and air just maintains it; jumps are forced+speed-neutral at 832. The high-speed finish is DOWNHILL-TERMINAL-VELOCITY limited (1544). So the only remaining levers are: (1) reach the downhill/high-speed sections marginally EARLIER (sub-pixel PHASE -- the search keeps sub-pixel in the hash and probes this); (2) RESOLVED-DEAD: a higher downhill terminal velocity is NOT input-reachable; (3) the jump7 1344->1288 step is NOT a penalty -- it is velX tracking a slope change.

**THROTTLE RULE (decisive, from tas.ram by section)**: the accel routine sub_CE29 HARD-CLAMPS velX DOWN to the cap (CE4D forces velX=tbl_C0CE/C0D1[Y] whenever it exceeds it) -- but ONLY runs when an A/B button is held (a throttle Y is selected). So: on the FLAT the TAS holds **B** -> pinned at 832 cap (631/1063 B + heat-coast). On DOWNHILLS the TAS **COASTS** (1063-1196: 125 coast vs 8 B) -> with NO button the clamp never runs, so GRAVITY pushes velX past 832 freely to each slope's TERMINAL velocity (832 flat -> 1088 -> 1288 -> 1344 -> **1544** steepest, held 273f). Holding B on a downhill would CLAMP velX back to 832 (a trap) -- the TAS correctly coasts. velX past the flat cap is PURELY slope(track-geometry)-determined + input-independent. So there is NO throttle/speed slack anywhere: flat=hard-capped, downhill=coast-to-terminal, air=maintains. The per-frame-airborne search optimizes arc+landing-angle+lean and TIES at 1468; the ONLY sub-frame slack left is sub-pixel PHASE on slope-boundary crossings (search-limited, still probing -- per [[never-call-tas-optimal]] NOT optimal, "no macro beat lever found"). Don't chase throttle/temperature/terminal-velocity. See [[excitebike-tas-beat]].
